```vue
<template>
  <form @submit.prevent="submitMentorshipRequest" class="mentorship-form">
    <h2>Request a Mentor</h2>

    <label for="goals">Goals (max 500 chars)</label>
    <textarea
      id="goals"
      v-model="form.goals"
      rows="4"
      maxlength="500"
      required
    ></textarea>

    <label for="availability">Availability (e.g., Mon‑Wed 10‑12)</label>
    <input
      id="availability"
      v-model="form.availability"
      type="text"
      required
    />

    <label for="expertise">Desired Expertise</label>
    <input
      id="expertise"
      v-model="form.expertise"
      type="text"
      placeholder="e.g., Front‑end, Data Science"
      required
    />

    <button type="submit" :disabled="submitting">
      {{ submitting ? 'Submitting…' : 'Submit Request' }}
    </button>
  </form>
</template>

<script>
export default {
  data() {
    return {
      form: {
        goals: '',
        availability: '',
        expertise: '',
      },
      submitting: false,
    };
  },
  methods: {
    async submitMentorshipRequest() {
      this.submitting = true;
      try {
        const payload = { ...this.form, userId: this.$store.state.user.id };
        const res = await this.$axios.post('/api/mentorship-requests', payload);
        this.$toast.success(res.data.message);
        this.form = { goals: '', availability: '', expertise: '' };
      } catch (err) {
        const msg = err.response?.data?.error || 'Submission failed';
        this.$toast.error(msg);
      } finally {
        this.submitting = false;
      }
    },
  },
};
</script>
```

## 1. Acceptance Criteria
1. **Form rendering** – The MentorshipRequestForm component displays input fields for *Goals* (text area, required) and *Availability* (select list, required) and a *Submit* button.
2. **Client‑side validation** – The form prevents submission when either field is empty and shows an inline error message.
3. **Request submission** – On valid submit, the front‑end POSTs a JSON payload `{ goals, availability }` to `/api/mentorship/request` and shows a “request received” toast.
4. **Automatic matching** – The back‑end service `MentorMatchingService.match(request)` returns a mentor whose expertise overlaps with the submitted goals **and** whose availability overlaps with the requested slots.
5. **Email notifications** – After a successful match, the system sends two separate email messages (via the existing mailer): one to the requester and one to the matched mentor, each containing the counterpart’s name, contact email, and agreed availability.
6. **Idempotent handling** – If the same user submits an identical request within 5 minutes, the API returns HTTP 200 with the previously created match instead of creating a duplicate entry.
